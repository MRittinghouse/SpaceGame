## 1. Scope and Relationship to Existing `PoliticsManager`

### 1.1 What the existing `PoliticsManager` does

`spacegame/models/politics.py` -- `PoliticsManager` -- handles ambient inter-faction political events. These are `PoliticalEvent` instances surfaced through dialogue trees: the player encounters an event (TRADE_DISPUTE, BORDER_INCIDENT, AID_REQUEST, DIPLOMATIC_SUMMIT, SANCTION, PIRATE_CRISIS) and chooses one of four `PoliticalAction` responses: SIDE_WITH_A, SIDE_WITH_B, MEDIATE, IGNORE.

This is ambient world texture. The player reacts to events that happen between factions. These events are generated by `PoliticsManager._generate_event()`, driven by game-day ticks, and resolved in a single interaction. They are not scheduled, not multi-round, and not associated with a specific named venue.

### 1.2 What the new dispute system does

The new dispute system is venue-based deliberation. The player enters a specific location -- Mayors' Council Chamber (Verdant), Alliance Congress Hall (Haven's Rest), Wreckers' Guild arbitration floor (Crimson Reach) -- where active disputes are scheduled, delegates are named and voiced, and the player influences the outcome through structured argument construction, coalition building, and mediation.

This is active political engagement. The player participates in deliberation, not just reacts to world events. Disputes persist across game sessions, have named delegates with distinct positions, and resolve to outcomes with concrete market and reputation consequences.

### 1.3 Coexistence decision

**The two systems coexist. Decision locked. See Section 11, decision 1.**

The existing `PoliticsManager` and its `PoliticalEvent` / `PoliticalAction` infrastructure remain unchanged. The new dispute system introduces: `PoliticsDispute`, `PoliticsDelegate`, `PoliticsArgument`, and a new manager class `PoliticsDisputeManager`.

These live in a new module: `spacegame/models/politics_dispute.py`.

Rationale: the two systems address different fictional surfaces. Ambient events are world events that occur regardless of player presence. Venue disputes are structured deliberations that require player presence at a specific location. Merging them would produce a base class that serves neither surface well.

### 1.4 Shared primitives (reused by the new system)

The new dispute system reuses these existing primitives:

| Primitive | Source | Usage |
|---|---|---|
| `apply_reputation_with_spillover(player, faction_id, amount)` | `PoliticsManager` (line 399) | All rep deltas from dispute outcomes |
| `SocialManager.get_effective_level(skill_id, npc_id)` | `social.py` (line 249) | Base skill level for argument checks |
| `SocialManager.resolve_check(skill_id, difficulty, npc_id)` | `social.py` (line 308) | Discrete pass/fail for corridor coalition visits |
| `crew_roster.get_bonus("bonus_type")` | crew system | Crew bonus stacking (matching `mining_view.py:417`) |
| `progression.get_bonus("bonus_type")` | progression system | Skill tree bonus stacking |
| `player.dialogue_flags` | player model | Tip flags and mission unlock flags |

### 1.5 Primitives NOT used by the new system

| Primitive | Reason not used |
|---|---|
| `PoliticalEvent` instances | Ambient events; not venue disputes |
| `PoliticalAction.SIDE_WITH_A / SIDE_WITH_B / MEDIATE / IGNORE` | Single-choice ambient actions; the new system uses vote/argue/mediate/abstain/coalition-build with separate mechanics |
| `PoliticsManager._relationships` / `modify_relationship()` | Ambient faction-to-faction tracking; disputes use `apply_reputation_with_spillover` directly |

Note on `PoliticalAction.MEDIATE`: this enum value stays in the codebase for backward compatibility with saved dialogue choices. The "mediate" player input mode in venue disputes is a separate concept with argument-construction mechanics. It does not use `PoliticalAction.MEDIATE`.

### 1.6 Module layout

```

spacegame/models/politics_dispute.py   -- NEW (SA-P2): PoliticsDispute,
                                          PoliticsDelegate, PoliticsArgument,
```

## 2. Dispute Lifecycle and Round Structure

### 2.1 Round count

- Standard disputes: **3 rounds**.
- Campaign-arc disputes (multi-session narrative arcs, SA-P3): **5 rounds**.

Round count is a field on the dispute template (`round_count: int`). There is no magic constant in view code. SA-P3 content authors set `round_count` per template.

### 2.2 Phase order per round

Each round has four phases, executed in order:

1. **Open arguments**: The player chooses one action: argue (opens argument composer, section 5.2), mediate (opens argument composer in mediation mode, section 5.3), abstain (section 5.4), or vote (ends dispute immediately, section 5.1). One player action per round.

2. **Counter-arguments**: Each delegate whose `visible_state` is `leaning_yes` or `wavering` fires one counter-argument. Counter-arguments target the player's argument's audience slot. If the player pre-loaded a `responds_to` framing in their argument's optional slot, the counter-argument is pre-empted for that round. Pre-emption prevents the state regression described in section 4.4.

3. **Conviction adjustments**: Delegate `visible_state` values update deterministically based on the outcome of phases 1 and 2. Hidden position vectors adjust by fixed amounts. No randomness at this phase.

4. **Vote-or-defer**: If this is the final round OR the player voted in phase 1, the vote tallies. Otherwise the dispute defers. Deferred disputes persist and resume at the player's next venue visit.

### 2.3 Game-day deadline

Each dispute has a `closes_on_day: int` field. If the player has not voted or abstained by the game day equal to `closes_on_day`, abstain is registered automatically and the dispute resolves with the current delegate state.

`closes_on_day` is calculated at dispute creation: `current_game_day + template.deadline_days`.

Defaults: 10 game days for standard disputes, 20 for campaign-arc disputes. SA-P3 content templates set `deadline_days` per dispute.

### 2.4 Multi-session persistence

In-progress disputes are stored in `player.politics_state.pending_disputes: list[PoliticsDispute]`. The venue reads this list on entry and surfaces active disputes. Completed disputes move to `player.politics_state.resolved_disputes: list[PoliticsDispute]` with outcome recorded.

When the player leaves the venue mid-dispute, the dispute serializes at the round boundary. Mid-round actions (clicking through the argument composer, reviewing delegate state before submitting) are not saved. The dispute resumes at the start of the next uncommitted round on re-entry.

### 2.5 Dispute lifecycle state diagram

```
                     [CREATED]
                         |
                  venue entry / re-entry
                         |
               +---[ROUND_OPEN]----+
               |                  |
          player votes         round ends
          before end           (no vote)
               |                  |
         [RESOLVING]         [ROUND_PENDING]
               |                  |
          outcome             next venue visit
          matrix                   |
          fires               +----+
               |              |
         [RESOLVED]      round < round_count?
                               |
                          Yes: next ROUND_OPEN
                          No:  force abstain -> RESOLVING

     deadline reached at any point -> abstain -> RESOLVING
```

### 2.6 Save / load granularity

**Round boundary.** The dispute `to_dict()` / `from_dict()` serializes the committed state after each round ends. SA-P2 implements these methods on `PoliticsDispute`. New fields added after SA-P2 ships must use `data.get("field", default)` in `from_dict()` per CLAUDE.md conventions (section 11, decision 15).

## 3. Dispute Templates and Variations

### 3.1 Template schema

Each dispute template is a frozen dataclass loaded by DataLoader from `data/politics/`. Required fields:

| Field | Type | Example |
|---|---|---|
| `id` | `str` | `"water_rights_phasing"` |
| `headline` | `str` | `"Water Rights Phasing Bill"` |
| `factions_affected` | `list[str]` | `["verdant", "frontier_alliance", "crimson_reach"]` |
| `base_difficulty` | `int` (1-5) | `4` |
| `round_count` | `int` | `3` |
| `deadline_days` | `int` | `10` |
| `delegate_roster` | `list[str]` | `["ferron_hask", "samela_drift", "ollo_marsh"]` |
| `eligible_framings` | `list[str]` | `["data_precedent", "soil_impact", "frontier_autonomy", "practical_cost", "community_benefit"]` |
| `eligible_evidence` | `list[str]` | `["forgeworks_2324_partnership", "verdant_soil_survey_2330", "northern_aquifer_draw_report"]` |
| `delegate_positions` | `dict[str, dict[str, float]]` | See section 4.1 |
| `outcome_matrix` | `dict[str, OutcomeRow]` | See section 3.2 |
| `is_campaign_arc` | `bool` | `False` |
| `required_flags` | `list[str]` | `[]` |

All fields are required. No template may omit any of them. "TBD" is not a valid field value.

### 3.2 Outcome matrix schema

Four outcome categories: `win`, `partial_win_coalition_thin`, `partial_win_off_record`, `loss`. Each maps to an `OutcomeRow`:

| Field | Type | Notes |
|---|---|---|
| `rep_deltas` | `dict[str, int]` | Faction ID to rep delta |
| `market_shifts` | `list[MarketShift]` | See section 3.3 |
| `mission_unlocks` | `list[str]` | Flag helper strings |
| `mission_locks` | `list[str]` | Flag helper strings |
| `news_headline` | `Optional[str]` | Required for win/loss; see section 7.6 for partial-win thresholds |

### 3.3 MarketShift schema

| Field | Type | Example |
|---|---|---|
| `commodity_id` | `str` | `"fresh_water"` |
| `system_id` | `str` | `"verdant"` |
| `magnitude` | `float` | `0.12` (fractional price multiplier) |
| `duration_days` | `int` | `30` (locked; see section 7.2) |

Magnitude range: +/-0.05 to +/-0.15 (5% to 15%). Duration: 30 game days, always.

## 4. AI Delegate Behavior Model

### 4.1 Hidden position vector

Each delegate has a hidden position vector defined per dispute template in `delegate_positions`. The dimensions are dispute-specific (a water-rights dispute has different axes than a trade-access dispute). Standard practice for Verdant disputes: 3 dimensions.

For water_rights_phasing:

| Dimension | Description | Range |
|---|---|---|
| `modernization` | Stance on modernization proposals | -1.0 (strongly opposed) to +1.0 (strongly in favor) |
| `water_rights_change` | Willingness to change water allocation | -1.0 (strongly opposed) to +1.0 (strongly in favor) |
| `outside_influence` | Receptivity to external parties and evidence | -1.0 (resistant) to +1.0 (open) |

Starting positions for water_rights_phasing delegates (defined in the dispute template's `delegate_positions` field):

| Delegate | modernization | water_rights_change | outside_influence |
|---|---|---|---|
| Ferron Hask | -0.8 | -0.7 | -0.6 |
| Samela Drift | +0.7 | +0.3 | +0.5 |
| Ollo Marsh | 0.0 | -0.9 | -0.2 |

These values are hidden from the player unless Cass Weller is on crew (section 7.5). The player discovers them through argument outcomes and delegate reaction lines.

Alliance Congress disputes (SA-P4) use different dimension labels suited to congressional-scale disputes (e.g., `frontier_autonomy_stance`, `trade_leverage`, `process_fidelity`). SA-P4 content authors define their own dimension labels in their templates.

### 4.2 Persuasion vectors

Each framing category targets a position dimension. A successful argument using a framing shifts the targeted delegate's relevant dimension by +0.20 (per successful argument in that round). The framing also contributes a modifier to the argument's effective level.

| Framing ID | Display name | Primary target dimension | Framing modifier |
|---|---|---|---|
| `data_precedent` | Data Precedent | `modernization` | +1 |
| `soil_impact` | Soil Impact | `water_rights_change` | +1 |
| `frontier_autonomy` | Frontier Autonomy | `outside_influence` | +1 |
| `practical_cost` | Practical Cost | all (low effect) | 0 |
| `community_benefit` | Community Benefit | `outside_influence` | 0 |

Framings work better against delegates who are receptive in the relevant dimension. Hask's `outside_influence` of -0.6 means `frontier_autonomy` and `community_benefit` framings face headwind from disposition resistance when targeting him. The disposition modifier (section 4.3) captures this numerically.

The framing ID is the schema key. The display name is what the player sees in the argument composer.

### 4.3 Disposition modifier

Each delegate has a disposition value toward the player, tracked in the same system as `SocialManager` dispositions, starting at `DISPOSITION_DEFAULT` (50 per `social.py`).

The disposition modifier applied to argument resolution:

```
disposition_mod = (delegate_disposition - DISPOSITION_DEFAULT) // 10
```

This mirrors `SocialManager.get_effective_level()`. A delegate with disposition 60 contributes +1 to effective argument level; disposition 30 contributes -2. Disposition changes through successful arguments (small increase) and corridor coalition visits (moderate increase on success, small decrease on failure).

### 4.4 Deterministic per-round update rules

After each argument resolves, delegate states update by these fixed rules:

| Event | Effect on visible_state | Effect on position vector |
|---|---|---|
| Player argument passes | Target advances one step toward `leaning_yes` | Target dimension +0.20 |
| Player argument fails | No change | No change |
| Counter-argument (not pre-empted) | Target advances one step toward `leaning_no` | Target dimension -0.15 |
| Counter-argument (pre-empted) | No change | No change |
| Mediation success | Target gains `conceded` flag (not visible_state) | Target dimension +0.10 |
| Mediation fail | No change | No change |

Visible state progression chain (in order):
```
committed_no -> leaning_no -> wavering -> leaning_yes -> committed_yes
```

A `committed` delegate (either direction) cannot be moved by arguments. They can be reached by corridor coalition visits (section 5.5) before the session, but `committed` states are rare at the start of round 1. Delegate positions reaching +-1.0 are capped at +-1.0.

Starting visible states for water_rights_phasing:

| Delegate | Starting visible_state |
|---|---|
| Ferron Hask | leaning_no |
| Samela Drift | wavering |
| Ollo Marsh | leaning_no |

### 4.5 Bias values

Each delegate in the dispute template has two bias fields:

| Bias field | Type | Effect |
|---|---|---|
| `faction_loyalty` | float (0.0-1.0) | Scales resistance to framings that oppose the delegate's faction mandate. Applied at session initialization as a position-vector adjustment toward the faction's known stance. Hask: 0.7; Drift: 0.4; Marsh: 0.8. |
| `prior_dispute_memory` | int (-1 = none, or rounds-ago count) | If > 0 and the player had a favorable outcome in a prior dispute at this venue, grants +5 to starting disposition toward the player for delegates who were favorable in that prior dispute. |

Bias values are applied once at session initialization, before round 1 begins.

### 4.6 Worked example: Hask and Drift across three rounds of water_rights_phasing

This is the SA-P2 implementer's fixture spec. Write unit tests against this example.

**Pre-session state (after bias initialization):**

```
Hask:  position {modernization: -0.8, water_rights_change: -0.7, outside_influence: -0.6}
       visible_state: leaning_no
       disposition: 50

Drift: position {modernization: +0.7, water_rights_change: +0.3, outside_influence: +0.5}
       visible_state: wavering
       disposition: 50

Marsh: position {modernization: 0.0, water_rights_change: -0.9, outside_influence: -0.2}
       visible_state: leaning_no
       disposition: 50
```

Player configuration: Persuasion 3, Desta Coll on crew (coalition_sway_bonus +0.15), coalition_sway skill at L2 (+0.20). No Cass Weller.

**Round 1 -- Player argues to Drift, framing: data_precedent, evidence: forgeworks_2324_partnership:**

```
effective = 3 + 1 (data_precedent) + 0 (Drift disposition 50) + 0.15 (Desta) + 0.20 (skill L2)
          = 4.35 -> floor to 4
base_difficulty = 4
4 >= 4 = pass
```

Drift visible_state: wavering -> leaning_yes.
Drift position modernization: +0.7 + 0.20 = +0.90.

Counter-argument phase: Hask fires counter (soil_impact framing) targeting the argument's effect on Drift.
Player has no responds_to framing in slot one.
Counter not pre-empted.
Drift visible_state: leaning_yes -> wavering.
Drift position water_rights_change: +0.3 - 0.15 = +0.15.

Conviction adjustments: Hask = leaning_no (unchanged), Drift = wavering, Marsh = leaning_no.

**Round 2 -- Player argues to Hask, framing: soil_impact, evidence: verdant_soil_survey_2330, responds_to: data_precedent in optional slot:**

```
effective = 3 + 1 (soil_impact) + 0 (Hask disposition 50) + 0.15 (Desta) + 0.20 (skill)
          = 4.35 -> floor to 4
4 >= 4 = pass
```

Hask visible_state: leaning_no -> wavering.
Hask position water_rights_change: -0.7 + 0.20 = -0.50.

Counter-argument phase: Hask fires counter (data_precedent framing) targeting Drift's wavering state.
Player pre-loaded responds_to: data_precedent in optional slot.
Counter pre-empted for this round.
Drift: no change, stays wavering.

Conviction adjustments: Hask = wavering, Drift = wavering, Marsh = leaning_no.

**Round 3 -- Player votes:**

Vote tally using visible_state-to-vote mapping (section 5.1):
- Drift (wavering) = votes no.
- Hask (wavering) = votes no.
- Marsh (leaning_no) = votes no.

Result: 0-3. Outcome: loss.
Outcome matrix fires: rep_deltas {verdant: -2, frontier_alliance: -1, crimson_reach: +2},
market_shifts (fresh_water -10% at verdant for 30 days), news_headline fires (loss).

**Alternate path (mediation in round 2):**

If instead of arguing to Hask in round 2, the player chose mediate targeting Marsh with arbitration_neutrality framing, a successful mediation check would set Marsh's `conceded` flag. If the vote then failed in round 3 (still likely with Hask and Drift only at wavering), the outcome would be `partial_win_off_record` rather than `loss`.

**Why this example matters for SA-P2:** It shows the distinction between winning the argument (moving visible_state) and winning the vote (tallying states at the end). The player advanced Hask from leaning_no to wavering across two rounds but did not complete the push to leaning_yes before voting, and did not secure Drift against Hask's counter. A player who understood the pre-emption mechanic and stocked responds_to: soil_impact in round 1 would have kept Drift at leaning_yes into round 3.

## 5. Player Input Model

Five modes. Coalition-build is pre-session. The other four are in-session.

### 5.1 Vote

**Mechanic**: Click "Vote" on the action bar. Choose yes or no.

**Skill check**: None.

**Cost**: Ends the dispute immediately. No further arguing or mediating is possible that session.

**Outcome**: Vote tallied with current delegate visible_states using this mapping:
- `committed_yes`: votes yes
- `leaning_yes`: votes yes
- `wavering`: votes no (not yet convinced)
- `leaning_no`: votes no
- `committed_no`: votes no

Simple majority of the delegate roster wins. Outcome category determined by:
1. Vote passes AND >= 60% of delegates were pre-committed via coalition-building: `win`
2. Vote passes AND < 60% pre-committed: `partial_win_coalition_thin`
3. Vote fails AND at least one delegate has `conceded` flag from a prior mediation: `partial_win_off_record`
4. Vote fails AND no mediated concessions: `loss`

**Trade-off**: Voting early forfeits remaining argument rounds but avoids counter-arguments. This matters when the coalition is secure and further rounds carry more risk than reward.

### 5.2 Argue

**Mechanic**: Click "Argue". Opens the argument composer (section 6). Submit framing + evidence + audience. One argue action per round.

**Skill check**: Multi-component effective argument level vs `base_difficulty` (section 6.2). Deterministic.

**Cost**: Uses the player's action for the round. The player cannot also mediate that round.

**Outcome**: Pass moves the target delegate one step toward `leaning_yes` and shifts the targeted position dimension. Fail has no effect.

### 5.3 Mediate

**Mechanic**: Click "Mediate". Opens the argument composer in mediation mode. Select a target delegate. Choose framing (compromise category) and evidence. Audience is the selected delegate.

**Skill check**: Same formula as argue (section 6.2), but sources crew and skill bonuses from `arbitration_neutrality_bonus` (Cass Weller +0.15, mediation_instinct L1 +0.10 / L2 +0.20) instead of `coalition_sway_bonus`.

**Cost**: Uses the player's action for the round. Cannot also argue that round.

**Outcome**: Pass sets `conceded` flag on target delegate. This flag is not visible_state; it is a separate binary per delegate per dispute. If the final vote fails but any delegate has `conceded`, outcome is `partial_win_off_record` instead of `loss`. A passing vote is not blocked by mediation; if the vote passes, the outcome is still `win` or `partial_win_coalition_thin`.

Cass Weller on crew adds `arbitration_neutrality_bonus` (+0.15) to effective, making the mediation threshold easier to reach. `mediation_instinct` skill at L2 adds another +0.20. At full investment: 0.15 + 0.20 = +0.35 bonus before base skill.

### 5.4 Abstain

**Mechanic**: Click "Abstain" or let the game-day deadline pass.

**Skill check**: None.

**Cost**: Forfeits the round's action. Dispute advances to next round with no state changes.

**Outcome**: If the player abstains every round, the dispute reaches its final round and resolves as if the player voted no with whatever coalition is in place from corridor pre-commits only. This is usually `loss` unless coalition-building pre-committed a majority.

**Deadline abstain**: On the morning of `closes_on_day`, if the dispute is still in `ROUND_PENDING`, the system registers abstain and the dispute resolves with current state.

### 5.5 Coalition-Build (pre-session corridor)

**Mechanic**: From the venue's dispute list, the player can access the council corridor sub-screen before entering a session. Each delegate listed has a "Talk" option. The player chooses a framing approach and the system runs a social check.

**Skill check**: `social_manager.resolve_check("persuasion", difficulty, delegate_npc_id)`. Difficulty is the delegate's `base_difficulty` value. Discrete pass/fail.

**Cost**: Each corridor visit costs no game-day time. However, each failed corridor visit costs 1 reputation point with that delegate's sub-tier faction (SA-B-EXT-1 sub-rep). Repeated failures with the same delegate in successive sessions raise the corridor visit difficulty by +1 for the next session (resets after one successful visit or after the dispute resolves).

**Outcome**: A successful corridor visit sets that delegate's visible_state to `leaning_yes` at session start (overriding the starting visible_state in the template). Pre-committed delegates count toward the 60% threshold for `win` vs `partial_win_coalition_thin`.

**Pre-commit cap**:
```
max_precommit = 1 + floor(crew_roster.get_bonus("coalition_size_bonus")
                          + progression.get_bonus("coalition_size_bonus"))

Without investment:        1 + floor(0 + 0)     = 1
With Desta Coll only:      1 + floor(1.0 + 0)   = 2
With Desta Coll + skill L2: 1 + floor(1.0 + 1.0) = 3
```

On a 3-delegate dispute, reaching the 60% threshold (2 of 3 pre-committed) requires either Desta Coll alone or delegate_reach skill at L1. The base cap of 1 keeps the player one pre-commit short of the threshold without investment.

## 6. Argument-Construction Submechanic

### 6.1 Overview

Each argue or mediate action uses the three-slot argument composer. The player selects:

1. **Framing**: The rhetorical category. Drawn from `eligible_framings` in the dispute template.
2. **Evidence**: The concrete claim. Drawn from the player's accessible evidence pool (see section 6.5).
3. **Audience**: Which delegate the argument targets. Only delegates in `delegate_roster` are selectable.

A fourth optional slot exists for pre-emption:

4. **Responds-to** (optional): A framing category the player expects in Hask's or another delegate's counter-argument. If the counter-argument uses this framing, the counter is pre-empted.

### 6.2 Resolution formula

```
effective = base_skill + framing_mod + disposition_mod + crew_bonus + tree_bonus
```

Where:
- `base_skill` = player's Persuasion level (or Leadership for `frontier_autonomy` framing, per section 6.4)
- `framing_mod` = modifier from section 4.2 table (0 or +1)
- `disposition_mod` = (delegate_disposition - 50) // 10
- `crew_bonus` = `crew_roster.get_bonus("coalition_sway_bonus")` (Desta Coll: +0.15; 0.0 if not on crew)
- `tree_bonus` = `progression.get_bonus("coalition_sway_bonus")` (coalition_sway L1: +0.10; L2: +0.20)

Resolution: `floor(effective) >= base_difficulty` = pass.

For mediate mode, replace crew and skill bonus sources:
- `crew_bonus` = `crew_roster.get_bonus("arbitration_neutrality_bonus")` (Cass Weller: +0.15)
- `tree_bonus` = `progression.get_bonus("arbitration_neutrality_bonus")` (mediation_instinct L1: +0.10; L2: +0.20)

Base skill for mediate mode is always Persuasion (not Leadership).

### 6.3 Worked end-to-end example

**Scenario**: Persuasion 3, Desta Coll on crew (coalition_sway_bonus +0.15), coalition_sway at L2 (+0.20). Dispute: water_rights_phasing. Target: Drift (disposition 50). Framing: data_precedent (+1 framing_mod). Evidence: forgeworks_2324_partnership.

```
effective = 3 (Persuasion)
          + 1 (data_precedent framing_mod)
          + 0 (Drift disposition 50: (50-50)//10 = 0)
          + 0.15 (Desta Coll)
          + 0.20 (coalition_sway L2)
          = 4.35
floor(4.35) = 4
base_difficulty = 4
4 >= 4 = pass
```

Drift visible_state: wavering -> leaning_yes.
Drift position modernization: +0.7 + 0.20 = +0.90.

**Counter-argument scenario**: Hask fires counter with soil_impact framing. Player had no responds_to stocked. Counter not pre-empted. Drift visible_state: leaning_yes -> wavering.

**Pre-emption scenario**: If the player had put responds_to: soil_impact in the optional slot, Hask's counter would be pre-empted. Drift stays at leaning_yes.

**Second worked example (mediation with Cass Weller)**: Persuasion 3, Cass Weller on crew (arbitration_neutrality_bonus +0.15), mediation_instinct at L2 (+0.20). Target: Marsh (disposition 50). Framing: community_benefit (framing_mod: 0). Evidence: northern_aquifer_draw_report.

```
effective = 3 (Persuasion)
          + 0 (community_benefit framing_mod)
          + 0 (Marsh disposition 50)
          + 0.15 (Cass Weller)
          + 0.20 (mediation_instinct L2)
          = 3.35
floor(3.35) = 3
base_difficulty = 4
3 < 4 = fail
```

This mediation attempt fails. Marsh does not gain `conceded` flag. Without additional investment (higher Persuasion, better framing, or positive disposition), Marsh resists mediation. The player needs either Persuasion 4, a framing_mod +1 framing, or a favorable disposition modifier from a prior successful corridor visit to mediate Marsh at base_difficulty 4.

### 6.4 Skill routing by framing

| Framing | Base skill used |
|---|---|
| `data_precedent` | Persuasion |
| `soil_impact` | Persuasion |
| `frontier_autonomy` | Leadership |
| `practical_cost` | Persuasion |
| `community_benefit` | Persuasion |

`frontier_autonomy` uses Leadership because the framing is about authority and autonomy claims, not persuasion. SA-P2 routes the base_skill lookup by framing ID.

### 6.5 Evidence pool access

The player can only select evidence listed in `eligible_evidence` that they have unlocked. Access is granted by:
- Completing a mission that sets the evidence flag (e.g., completing "Forgeworks Partnership Survey" sets `evidence_forgeworks_2324_partnership`)
- Having faction standing >= 30 with the evidence's associated faction (unlocks faction-standing evidence)
- Owning specific commodities at the time of the session (unlocks commodity-ownership evidence)

If no eligible evidence is accessible, the evidence slot is empty and the argument is composed as "framing + no evidence + audience." Absent evidence increases effective difficulty by +1 (evidence provides grounding; without it the argument is less anchored).

### 6.6 Slot dependency

| Slot | Required? | Empty slot penalty |
|---|---|---|
| Framing | Yes | Argument cannot be submitted without framing |
| Evidence | No | +1 to effective difficulty if absent |
| Audience | Yes | Argument cannot be submitted without target selection |
| Responds-to | No | Counter-arguments are not pre-empted if absent |

## 7. Integration Commitments

### 7.1 Reputation

All rep deltas from dispute outcomes flow through `apply_reputation_with_spillover(player, faction_id, amount)` from `PoliticsManager` (line 399 of `politics.py`). Each entry in `outcome_matrix[category].rep_deltas` is a separate call.

Example for water_rights_phasing win:
```python
politics_manager.apply_reputation_with_spillover(player, "verdant", +5)
politics_manager.apply_reputation_with_spillover(player, "frontier_alliance", +2)
politics_manager.apply_reputation_with_spillover(player, "crimson_reach", -2)
```

The spillover pipeline applies rival penalties automatically. Rep deltas are calibrated so a single dispute win moves the player approximately half a tier step in the primary faction.

Typical rep delta ranges by outcome category:

| Outcome | Primary faction | Rival faction |
|---|---|---|
| win | +3 to +6 | -1 to -3 |
| partial_win_coalition_thin | +1 to +3 | -1 |
| partial_win_off_record | +2 to +4 | 0 |
| loss | -1 to -3 | +1 to +3 |

### 7.2 Market

Dispute outcomes shift commodity prices at affected systems for a fixed window.

**Duration**: 30 game days. Locked. SA-P3/P4/P5 content templates must use 30 for market shift duration.

**Magnitude range**: +/-5% to +/-15% (0.05 to 0.15 as fractional multiplier). Locked.

Implementation: SA-P2 adds a time-based market shift record to the market model. On each game-day tick, the market tick handler applies or expires active shifts. The larger of two overlapping shifts for the same commodity at the same system applies; shifts do not stack (section 11, decision 14).

Sample matrix by dispute category (for SA-P3 content authors):
- Water-rights disputes: fresh_water (primary), hydroponics_yield (secondary)
- Modernization disputes: industrial_components (primary), construction_materials (secondary)
- Trade-access disputes: foodstuffs, refined_metals (at multiple systems in the faction region)

Win shifts commodity prices in favor of the winning position. Loss shifts in the opposing direction.

### 7.6 News

A dispute outcome fires a news ticker entry when ALL conditions hold:

**Condition A** (at least one of):
- The outcome shifts a faction's reputation across a tier boundary (tier thresholds: -75, -25, +25, +75)
- The outcome shifts a tracked commodity price by >= 10% at the affected system

**Condition B**:
- The outcome category is `win` or `loss`

Partial wins (`partial_win_coalition_thin`, `partial_win_off_record`) do NOT generate news unless condition A is also met (tier boundary crossing). This prevents routine partial wins from flooding the ticker.

News headline format:
- One line, maximum 80 characters
- Third-person, present-tense impact
- No em-dashes; semicolon separates event from consequence
- Register: factual, ticker-brief, not editorializing

Format template for SA-P3/P4/P5 content authors:
```
"[Venue/council] [action] [subject]; [consequence note]."
```

Sample compliant headlines (character counts verified):
- "Verdant council phases water rights; hydroponics shift expected, 30 days." (74 chars)
- "Verdant council rejects water rights phasing bill; farmers' bloc holds." (71 chars)
- "Alliance Congress passes frontier trade accord; Haven's Rest market impact." (75 chars)

## 11. Decisions Locked and Questions Explicitly Deferred

### Locked decisions

1. **PoliticsManager coexistence, not merge.** The existing `PoliticsManager` and the new `PoliticsDisputeManager` coexist. They share only the `apply_reputation_with_spillover` pipeline. Rationale: ambient political events and venue disputes are different fictional surfaces. A shared base class serves neither well. The existing `PoliticalAction.MEDIATE` stays for backward compatibility.

2. **New module `politics_dispute.py`.** `PoliticsDispute`, `PoliticsDelegate`, `PoliticsArgument`, `PoliticsDisputeManager` all live in `spacegame/models/politics_dispute.py` (a new file, SA-P2). Rationale: keeps `politics.py` clean; SA-P2 can test the new module in isolation without touching ambient event logic.

3. **Player-paced rounds with game-day deadline.** No real-time clock within a round. The player composes arguments and commits when ready. `closes_on_day` enforces stakes without pressuring per-decision reading speed. Rationale: real-time clocks conflict with the "deterministic outcomes" axiom in CLAUDE.md. Save scumming with a deadline system is prevented because the deadline auto-resolves rather than pausing.

4. **Round-boundary save granularity.** The dispute serializes committed state at the start and end of each round. Mid-round actions are not saved. Rationale: action-boundary saves leak partial argument-composer state into save files, complicating round-trip serialization. Session-boundary saves lose mid-dispute progress. Round boundaries are clean commit points aligned with the per-round deterministic delegate update rules.

5. **Deterministic partial-win categories.** Four outcomes: `win`, `partial_win_coalition_thin`, `partial_win_off_record`, `loss`. No probability rolls at resolution. The 60% coalition-thin threshold is the numerical boundary. Rationale: CLAUDE.md states "social skill checks use threshold comparison (effective_level >= difficulty), NOT random rolls. No save scumming. Skills and investment determine success, not luck."

6. **Explicit pre-session corridor for coalition-building.** The player visits delegates in the council corridor before a session opens. Each visit is a social skill check. Pre-committed delegates start at `leaning_yes`. Rationale: corridor work matches the character register ("The visible vote is the last thing that matters" -- Desta Coll, `character_voices.md:1545`). It gives `coalition_size_bonus` and `delegate_reach` mechanically meaningful surfaces.

7. **Tutorial fold-in lives in SA-P3.** SA-P1 specifies tip flag names and draft tip text. SA-P3 wires them via `FirstTimeTipOverlay`. Rationale: the same pattern used in SA-V (tip wired by the venue sprint, not a separate sprint). SA-X3 handles cross-anchor tutorial polish.

8. **News threshold = (tier-crossing OR >= 10% commodity shift) AND (win or loss).** Partial wins do not generate news unless they cross a tier boundary. Rationale: every dispute generating news dilutes the ticker. Gated headlines preserve signal and make ticker entries feel like meaningful events in the world.

9. **Verdant and Alliance Congress voice sheets sufficient.** The four Verdant voices (Vance, Hask, Drift, Marsh) and four Congress voices (Wentworth, Shirane, Vasc, Tejada) in `character_voices.md` cover their respective venues without rework. Additional per-dispute named delegates in SA-P3/P4 content are handled as one-off characters by the content authoring sprint. Rationale: SA-PREP-1 explicitly authored these sheets for this purpose.

10. **AI worked example fixture: Hask and Drift on water_rights_phasing.** The three-round example in section 4.6 is the SA-P2 implementer's unit test fixture specification. Rationale: Hask's high resistance to modernization and Drift's data-driven counter-move (pre-empting Hask, per `character_voices.md:987`) give the clearest observable contrast for testing the persuasion vector model.

11. **Evidence pool gated by mission completion, faction standing, commodity ownership.** An empty evidence slot increases effective difficulty by +1. Rationale: this makes mission completion and faction investment feel like they directly unlock better arguments, which reinforces "progression depth" from the gameplay philosophy.

12. **Mediation uses `arbitration_neutrality_bonus`, argue uses `coalition_sway_bonus`.** The two crew specialists (Desta Coll / Cass Weller) apply to distinct modes. Neither applies to both. Rationale: keeps the crew roles mechanically distinct. A player with Desta but not Cass is stronger in argue/coalition modes; with Cass but not Desta, stronger in mediation. Both together give the full toolkit.

13. **Wavering delegates vote no.** A delegate at `wavering` visible_state at vote time votes no. Rationale: "wavering" means they have been moved from firm opposition but are not yet convinced. The player must push through to `leaning_yes` for a guaranteed yes vote. This gives the argument composer real stakes in each round.

14. **Market shifts do not stack.** When two active market shifts apply to the same commodity at the same system, the larger absolute magnitude applies. Rationale: stacking would allow indefinite price manipulation through repeated dispute wins. The "largest active shift" rule preserves market shifts as a temporary consequence window, not a permanent lever.

15. **`from_dict()` on `PoliticsDispute` must use `data.get("field", default)` for all fields.** Any field added after SA-P2 ships must follow this pattern. Rationale: per CLAUDE.md, `from_dict()` must handle missing keys for backward compatibility with older saves.
